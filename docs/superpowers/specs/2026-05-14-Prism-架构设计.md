| 层 | 模块 |

## 二、模块详解

### 2.1 Collector · 采集

**职责**：从 Twitter 持续抓取相关推文池

**输入**：
- 用户 KOL 关注列表（质量信号 / 扫描种子）
- 扩展扫描策略（参数化）

**输出**：原始推文流

**核心约束**：
- **扫描范围 ≫ KOL 列表** — 必须主动扩展到 KOL 网络（被 KOL 关注 / 提及 / 转推的人）+ 话题相关讨论者。否则违反 spec 失败模式 3「反茧房失败」
- 不做实时流（v1 定时抓取即可）
- v1 不做跨平台（spec 本期克制）

**v2 钩子**：接收 SignalFeedback 调整扫描权重 / 扩展方向

### 2.2 Translator · 翻译

**职责**：英文 / 其他语言推文翻译成中文

**输入**：原始推文
**输出**：含 `text_zh` 字段的推文（原文保留 fallback）

**质量要求**：翻译错误反而误导（spec 失败模式 5）— 应在翻译输出端做合理性校验或保留原文链接

### 2.3 Topic Clusterer · 话题聚类

**职责**：把推文按"讨论的话题"聚类

**输入**：翻译后的推文池
**输出**：话题对象 + 推文 → 话题的归属关系

**核心约束**：
- 工具自动聚类（不预设话题，spec 第 4 节）
- 颗粒度自适应 — 不强制聚类数

**Anti-pattern**：
- 把不同事件合并成同一话题
- 把同一事件拆成多个话题
- 这两类错误都很难看（spec 失败模式 6「立场聚类准不准」的前置）

### 2.4 Stance Analyzer · 立场分析

**职责**：在每个话题内识别立场，把推文分到立场组

**输入**：话题 + 该话题下的推文
**输出**：话题（含 stance groups）+ 每条推文的 stance 归属

**立场色映射**（详见 UX 设计文档第 2.1 节）：
- `bullish` · 看多
- `bearish` · 看空
- `skeptical` · 质疑数据
- `macro_bearish` · 宏观偏空
- `neutral` · 中性
- `contrarian` · 反主流

**核心约束**：
- 立场数量**动态**（每个话题可能 2-6 种）
- 立场标题要**短而能表达意图**（UX 第 4.3 节）
- 立场不依赖说话人身份（不分圈内/外，spec v0.3 修订）

### 2.5 Importance Ranker · 重要度排序

**职责**：AI 综合判断每个话题的"重要度 / 热度"

**输入**：话题（含 metadata）
**输出**：含 `importance_score` 的话题

**输入信号**（AI 综合，**不**机械加权）：
- 推文数量 / 参与人数
- 立场多样性（分歧度）
- 高质量来源参与度（KOL 列表内的人 = 质量信号）
- 跨圈共振（多个类别都在讨论 — Trend Monitor 输出反馈）
- 时间衰减
- v2: 历史观点验证准确率（来源权重）

**反信号**（必须避免）：
- 单纯依赖互动量（防 spec 失败模式 4「不能区分热度与质量」）

### 2.6 Category Inducer · 类别归纳

**职责**：从话题集合中**动态归纳**出"类别"

**输入**：话题集合
**输出**：动态类别列表 + 话题 → 类别映射

**核心约束**：
- **不预设类别**（spec v0.3 修订：去掉 crypto/AI 限定）
- 类别由内容内容决定
- 类别数大致 5-10 个（太多则降低）

### 2.7 Trend Monitor · 趋势监测

**职责**：检测圈层级核心热度变化

**输入**：话题流 + 历史话题状态
**输出**：赛道趋势列表（名字 / 变化幅度 / 描述）

**关键能力**：
- 跨话题、跨类别的"共振检测"
- 例：识别"crypto 都在说存储"（spec 失败模式 5 的反面）
- 时间窗对比（24h / 7d）

**输出消费方**：
- Frontend 右栏「赛道趋势」面板
- Importance Ranker 的输入（共振 = 重要度加成）

### 2.8 Store · 存储

**职责**：持久化所有数据

**子 store**：

| Store | 内容 | 备注 |
|---|---|---|
| TweetStore | 原始 + 翻译推文 | 历史保留（v2 验证用） |
| TopicStore | 话题 / 立场归属 / 重要度 | 含历史快照 |
| CategoryStore | 动态类别 | 周期性重建 |
| TrendStore | 赛道趋势历史 | 时序数据 |
| BookmarkStore | 用户收藏（推文层级） | first-class，v2 验证池输入源 |
| HistoryStore | v2 用，事件 / 验证记录 | v1 留空 |

**关键约束**：
- 所有"中间状态"持久化（不只是终态），便于调试和 v2 回溯
- Bookmark 是 first-class 数据（不只是 UI 标记）

### 2.9 API / Presenter · 呈现 API

**职责**：把 Store 数据按 UI 页面需求 expose

**输入**：前端查询
**输出**：页面所需数据

**主要接口**（按 UX 设计文档的页面 / 交互）：

| 接口 | 用途 |
|---|---|
| `listTopics(category?, time?, sort?)` | 主工作台话题列表 |
| `getTopic(id)` | 话题详情（含立场分组） |
| `listCategories()` | 左栏类别导航 |
| `listTrends()` | 右栏赛道趋势 |
| `addBookmark(tweet_id)` | 推文层级收藏 |
| `removeBookmark(tweet_id)` | 取消收藏 |
| `listBookmarks(filter?, sort?)` | 收藏列表页 |

### 2.10 Frontend · 前端 UI

详见 UX 设计文档。本架构层只关心 Frontend 与 API 的**接口契约**。

## 三、数据流（典型流程）

### 3.1 后台流水线（定期触发）

```
Step 1  Collector 抓取最近 N 时间窗的推文（含 KOL 网络扩展）
Step 2  Translator 翻译到中文
Step 3  Topic Clusterer 聚类成话题
Step 4  Stance Analyzer 在每个话题内分立场
Step 5  Trend Monitor 检测圈层共振
Step 6  Importance Ranker 综合打分（参考 Step 5 输出）
Step 7  Category Inducer 更新类别
Step 8  Store 持久化所有数据
```

### 3.2 用户交互流程

```
打开工作台  ─►  API.listTopics()
                + API.listCategories()
                + API.listTrends()
              └─►  Frontend 渲染主工作台三栏

点击话题卡  ─►  API.getTopic(id)
              └─►  Frontend 渲染详情页（立场分组）

点击收藏★  ─►  API.addBookmark(tweet_id)
              └─►  BookmarkStore 持久化
              └─►  v2: Signal Feedback Engine 接收信号

点击左栏「我的收藏」 ─► API.listBookmarks()
                       └─►  Frontend 渲染收藏列表
```

## 四、关键数据契约

不到具体序列化协议（JSON / protobuf 是技术层），只定义数据形状和必备字段。

### Tweet
```
{
  id, author, text, language,
  text_zh,           // 翻译（中文）
  posted_at,         // 发布时间 — v2 验证核心字段，不能省
  likes, views,      // 热度指标
  source_url,        // 原文链接
  collected_at       // 抓取时间
}
```

### Topic
```
{
  id,
  title_zh,
  category,                  // by Category Inducer
  importance_score,          // by Importance Ranker
  stance_distribution,       // { bullish: 3, bearish: 2, skeptical: 1, ... } 用于卡片色条
  stances: [StanceGroup],
  created_at, updated_at
}
```

### StanceGroup
```
{
  stance_label,      // 立场标题（短）："看多 · 周期信号"
  stance_type,       // bullish / bearish / skeptical / macro_bearish / neutral / contrarian
  tweet_ids: [...]
}
```

### Bookmark
```
{
  tweet_id,
  tweet_snapshot,    // 完整快照（含 posted_at + likes + views + text + text_zh）
  topic_id,          // 所属话题
  topic_snapshot_title,  // 话题标题快照（话题可能被归档）
  bookmarked_at
}
```

注：`tweet_snapshot` 是完整快照而非引用 — 因为 v2 需要"被验证池"，原推文可能被删 / 被改，但收藏内容必须保留。

### Category
```
{ name_zh, topic_count }
```

### Trend
```
{
  name_zh,
  change_pct,          // 24h 或 7d 涨跌
  sentiment_label,     // "主流共识：早期" / "立场分裂" / "多元观点"
  topic_ids: [...]     // 关联话题
}
```

## 五、v1 / v2 边界

### v1 范围
- 10 个核心模块的基础功能
- 完整流水线 + 前端
- 满足 spec v0.3 所有功能

### v2 增加模块
- **Signal Feedback Engine** — 隐式认可信号反馈
  - 输入：Bookmark store（用户收藏行为）
  - 输出：Collector 的扫描权重调整 + Ranker 的来源权重调整
  - 触发频率：定期 / 事件触发
- **Validation Engine** — 历史观点验证
  - 输入：Bookmark store（待验证观点池）+ 当前事件流
  - 输出：来源准确率评分 → 反馈给 Ranker
  - 这是产品长期差异化（spec 第 8.2 节）

### v1 必须预留的接口（防 v2 重做）
- `BookmarkStore`：字段完整（含 `tweet_snapshot.posted_at`）
- AI 调用通过统一 abstraction（便于 v2 加新调用 hook）
- 所有中间状态持久化（v2 回溯依赖）

## 六、架构原则

1. **模块独立** — 每个模块有明确 IO，可独立替换。例如 Translator 模块可以从"Claude 翻译"换成"Google 翻译" 而不影响其他模块
2. **接口契约优先** — 模块间通过定义良好的数据形状通信，不暴露内部状态
3. **持久化优先** — 所有中间状态持久化（不只是终态），便于调试 / 回溯 / v2
4. **AI 调用集中** — 所有 AI 调用通过统一抽象，便于切换模型 / 调参 / 加 hook
5. **反茧房不可妥协** — Collector 扫描范围必须 ≫ KOL 列表
6. **时间戳是数据公民** — 所有数据带时间戳（v2 验证机制依赖）
7. **Bookmark 是 first-class** — 不是 UI 标记，是产品核心数据

## 七、待用户决定的技术架构层（不在本文档范围）

以下问题由用户自己设计：

1. **技术栈**：编程语言 / Web 框架 / 前端框架（React / Vue / Svelte / 其他）

6. **任务调度**：cron / queue + worker / event-driven
7. **缓存策略**
8. **认证**（v1 单用户，可能不需要）

## 八、开放问题（架构层面）

1. **流水线触发频率**：每分钟 / 每 5 分钟 / 每小时？影响"打开即新"体验
2. **Tweet store 保留多久**：无限 / 30 天 / 90 天？影响存储成本和 v2 验证池规模
3. **Topic 生命周期 / 归档策略**：话题热度衰减到何时归档？归档后 Bookmark 引用怎么处理（已设计 `topic_snapshot_title`）
4. **后台流水线 vs 用户访问的同步**：用户打开时是否立即跑增量？还是只看上次定时跑的结果
5. **AI 调用的 batch 策略**：每条推文单独翻译 vs 批量翻译？影响成本和延迟
6. **Stance Analyzer 的"立场标签"复用**：同一立场标签（如"看多 · 周期信号"）在不同话题间是否复用？还是每个话题独立

## 九、给技术架构阶段的建议

虽然技术选型由用户决定，但本架构暗示了几个**有利于实现**的特征：

1. **模块化** — 各模块解耦适合微服务或单进程多模块都可
2. **管道化** — 处理层是单向管道，适合 stream 处理或 batch
3. **AI 重 IO 轻 CPU** — 主要瓶颈是 AI 调用次数和延迟
4. **存储多类型** — Tweet / Topic 是文档型；Bookmark 是关系型；Trend 是时序型 — 单库或多库都可
5. **前后端分离自然** — Frontend 通过 API 访问，可以本地或远端
