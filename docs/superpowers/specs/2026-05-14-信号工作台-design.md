# 信号工作台 — 产品设计文档（v0.3）

**日期**：2026-05-14
**状态**：需求 spec，未定技术架构（技术架构由用户自己设计）
**前序**：原 KOL Twitter 自动化项目已凉，本项目从复盘中重新立项

## 修订历史

- **v0.3 (2026-05-14)**：收藏机制升级
  - 收藏对象从「话题层级」改为「推文层级」（v1 就做，不是 v2）
  - 收藏数据持久化到本地，含发布时间等核心字段
  - 收藏 = 用户记忆库 = v2 "历史观点验证" 的直接输入源
  - 详情页推文卡片新增热度指标（点赞数、阅读量）
  - 立场标题风格明确为"短，但能表达意图"
- **v0.2 (2026-05-14)**：根本性修订
  - 删除「圈内 / 圈外」机制及对照圈面板（用户判定为不必要的复杂度）
  - KOL 关注列表角色从「分圈种子」改为「质量信号」
  - 反茧房机制重定义为「扩大扫描范围」（避开推荐算法茧房），不是「圈内圈外对照」
  - 详情页从「两栏对照」改为「按立场聚类的单页」
  - 右栏「对照圈面板」删除，「赛道趋势」独占右栏
  - 新增 v2 未来路线：认可信号反馈、历史观点验证（"现实给评分"）
- **v0.1 (2026-05-14)**：初版 spec

## 一、定位

> **一个中文信号工作台**：自动发现你会在 X 上关注的内容（不限定领域），用 AI 做翻译、聚类、热度判断；通过扩大扫描范围避开推荐算法茧房；让你不再需要打开 X。

### 成功标准
**让用户摆脱 X 平台。** 用户日常本来打开 X 消费内容的时间，全部迁移到这个工作台。

### 失败模式（必须主动规避）
1. 内容更新滞后
2. 热度信号失真
3. 内容池被锁在 KOL 关注列表里（即"反茧房失败"——仍然只看到推荐算法范围内的内容）
4. 不能在高热度中识别真高质量（"高知识密度"指深度 / 见解 / 信息量，不是单纯互动量）
5. 错过圈层级核心热度（例："crypto 都在讨论存储"，但工作台没感知到）
6. 信噪比低

## 二、不做（防需求爆炸）

注：用户的工作原则是"第一期克制 + 后续按需添加"。下面分两类：

### 2.1 本期克制（v1 不做，未来视使用反馈评估）

- 🟡 **跨平台**：v1 只 Twitter，Reddit / HN / 小红书 / 雪球 等留到 v2+
- 🟡 **AI 深度分析 / 解读 / 评论**：当前 AI 没有用户长期上下文，分析等于 Grok 水平没价值增量。如果未来 AI 有持久记忆能消化用户的长期视角，可以重新评估
- 🟡 **用户系统、多用户支持**：v1 是单用户工具
- 🟡 **主动推送 / 通知**：v1 不发通知，用户来访问工作台。未来如果需要"事件触发型紧急提醒"，可以加
- 🟡 **"自动改进系统"、参数调节 UI**：v1 不做自适应。等核心体验稳定后再评估

### 2.2 方向性排除（与工具定位冲突，原则上不做）

- ❌ **交易信号生成**：工具是"消费/校准"，不是"行动指令源"
- ❌ **内容创作 / 自动发推**：这是原 KOL 项目失败方向，本工具的输出方向是"读进来"，不是"写出去"
- ❌ **标注 / 打分 / 标记 UI**：用户角色是决策者 / 阅读者，不是数据工程师
- ❌ **领域限定**：工具的内容池开放，不锁定在 crypto / AI 等具体领域
- ❌ **「圈内 / 圈外」对照**：v0.1 的核心设计，v0.2 否定。用户关心的是观点本身，不是说话人的背景

## 三、核心理念

### 3.1 KOL 关注列表的角色 — 质量信号

用户 follow 的 KOL = 用户已经筛选过的高质量来源。

工具使用 KOL 列表的方式：
- 作为**质量信号 / 权重输入**：被 KOL 列表中人发布、引用、转发的内容获得更高权重
- 作为**扫描种子**：从 KOL 列表出发扩展到他们的网络（被他们关注的人、被他们引用的内容）

KOL 列表**不**用来：
- ~~区分"圈内 / 圈外"~~（v0.1 设计，已删除）
- ~~推断用户的"立场"或"圈层位置"~~（v0.1 设计，已删除）

### 3.2 反茧房机制 — 扩大扫描范围

反茧房的本质 = **避免陷入推荐算法给你的偏好内容**。

实现机制：工具的**内容池广度远大于用户的 KOL 关注列表**——爬 KOL 网络的扩展节点、爬话题相关的全部 Twitter 用户、不局限在用户的"信息泡泡"里。

反茧房是 v0.2 的核心理念之一，但**实现路径是"广度扫描 + 高质量筛选"**，不是"圈内圈外对照"。

### 3.3 AI 的角色

AI 是**筛选 / 整理 / 翻译 / 聚类**的工具，**不是**作者 / 分析师 / 评论员。

AI 在工作台里做的事：
- 翻译英文内容到中文
- 把推文按话题聚类
- 把同一话题下的推文按**立场 / 观点**聚类
- 综合判断话题"重要度 / 热度"
- 在广泛内容池里识别高质量内容
- 支持用户询问

AI 不做的事：
- 不在话题下生成"深度分析"段落
- 不替用户解读内容含义
- 不给观点
- 不评论

### 3.4 隐式输入

用户不需要显式写下"我现在认为 X"。工具通过：
- **KOL 关注列表**（质量信号 + 扫描种子）
- **用户在工作台里的隐式行为**（点击、点赞、收藏 — v2 才用）
- **历史观点验证**（v2 才用）

获取用户偏好和质量信号的输入。

## 四、核心体验

### 4.1 工作台整体布局（三栏式）

经过 brainstorm 确认采用 Layout A 修订版：

**左栏 · 类别导航（约 180px）**
- 主入口分组：📌 全部 / ⭐ 我的收藏
- 类别分组：工具自动发现的领域类别（Crypto / AI / 宏观 / 美股 / 科技 / 其他）
- 类别后面是当前话题数
- 类别是工具**自动发现**的（不预设、不限定，工具看到什么有什么）

**中栏 · 话题主区**
- 顶部筛选条：时间维度（今日 / 本周 / 本月）+ 排序维度（重要度 / 时间）
- 卡片列表：按所选排序展示话题

**右栏 · 赛道趋势面板**
- v1 阶段独占右栏（不再有"对照圈"面板）
- 显示当下整个圈在涨 / 在跌的话题（圈层热度感知）
- 架构上保留为「可扩展面板槽位」，未来可叠加更多面板（详见第八节）

### 4.2 话题卡片（中栏卡片元素）

**排序**：AI 综合判断的"重要度 / 热度"（不显示分数本身，让排序传达信息）

**每张卡片显示的预览信息**（按重要性排序）：
1. **时间**（最重要）—— 用户判断"是否值得点"的第一信号
2. **中文标题 / 一句话内容**（次要核心）—— 翻译过的话题摘要
3. *其他元素一律不放*（不放 KOL 头像、不放热度数字、不放 AI 摘要、不放参与者数量等）

设计原则：**卡片要克制**。

### 4.3 点开话题 → 详情页

**详情页采用"按立场聚类的单页"**（v0.1 的"左圈内 / 右圈外"两栏对比已删除）。

**顶部 bar**：← 返回 · 话题标题 · 时间 · 类别

（注：v0.2 顶部 bar 有「⭐ 收藏」按钮，v0.3 移除——收藏改为推文层级，按钮挪到每条推文卡片旁）

**主体**：
- AI 把这个话题下的推文按**立场 / 观点**聚类
- 每个立场是一个 group：
  - **立场标题**：短，但能表达意图。例："看多 · 周期信号"、"质疑数据"、"宏观偏空"、"短期事件驱动"。避免纯单词（"看多"信息量不足）和长句（"我们认为 ETF 资金流入..."冗长）
  - 该立场下的代表推文 2-5 条
- **立场排序**：AI 综合判断的"代表性 / 重要度"，立场内推文按时间或互动量降序
- **不区分说话人来源**——用户关心的是观点本身，不区分是否是 KOL 列表内/外的人

**每条推文卡片包含的字段**：
- 作者
- 发布时间（核心字段，是 v2 历史观点验证的关键）
- 中文翻译
- 原文链接
- **热度指标：点赞数 + 阅读量**（v0.3 新增 — 用户用来衡量推文热度的可见信号）
- **⭐ 收藏按钮**（v0.3 新增 — 推文层级收藏入口，详见第 4.5 节）

**不做**：
- 不做时间线演化视图
- 不做长篇 AI 摘要
- 不做"圈内 / 圈外"标签

### 4.4 赛道趋势面板（右栏）

显示当下整个圈在涨 / 在跌的话题——专门捕捉"圈层级核心热度"（失败模式 5 的反面）。

例：crypto 圈这两天集中讨论"存储赛道"，即使用户的 follow 列表里这条信号不明显，"赛道趋势"面板会让用户感知到。

形态：轻量列表，每个 entry 是一个"赛道 / 议题" + 趋势变化指标（涨 / 跌的程度），点击进入对应话题集合。

### 4.5 收藏机制（v0.3 大幅修订）

**收藏对象：单条推文**（推文层级，不是话题层级）

**收藏入口**：详情页中每条推文卡片旁的 ⭐ 按钮。点击 ⭐ → 推文持久化到本地。

**收藏列表入口**：左栏「主入口」分组下「⭐ 我的收藏」。

**收藏内容（持久化到本地的字段）**：
- 推文 ID / 原文 URL
- 原文内容
- 中文翻译
- 作者
- **发布时间**（核心字段，是 v2 历史观点验证的关键 — 不能省略）
- 收藏时间
- 所属话题（便于回看上下文）
- 热度数据快照（点赞数、阅读量在收藏时的值）

**收藏的定位**：
> **不只是"我喜欢这条"的标记，而是用户的"记忆库"——主动 curate 的、未来要回看 / 验证的观点池。**

**收藏页面（v1 阶段）**：
- 列出所有收藏的推文（默认按收藏时间倒序，可改按发布时间）
- 每条显示：原文 + 翻译 + 作者 + 发布时间 + 收藏时间 + 原文链接 + 所属话题
- 支持按时间筛选、按话题筛选（v1 基础筛选，v2 可加搜索）

**收藏的长期作用**（v2 详见第八节）：
- 8.1 认可信号反馈：收藏的来源 → 后续扫描权重提升
- 8.2 历史观点验证：**收藏池 = 验证池**——用户主动 curate 的"待验证观点"是历史验证机制最干净的输入源

## 五、使用节奏

**不定期看**。用户来的时间不固定，工具必须满足"打开即新"——任何时间打开，内容都是最新整理过的。

- 不强制 daily
- 不发通知
- 不要求 weekly review

## 六、待用户在实现阶段处理的硬问题（技术架构由用户决定）

以下不是 spec 能解决的问题，但**实现时必须有方案应对**：

1. **"热度 / 重要度" 怎么算才不失真？**
   - AI 综合判断（已确定方向）
   - 但具体输入哪些信号、怎么避免被互动量带偏、怎么识别"高知识密度"

2. **"圈层级核心热度" 怎么感知？**
   - 例如怎么让工具发现"crypto 都在讨论存储"
   - 涉及到"内容池广度 + 跨圈共振检测"

3. **"立场聚类" 准不准？**（新增 — v0.2）
   - AI 把推文按立场分组是详情页的核心，分错了会很难看
   - 立场标签怎么命名（短而准）
   - 边界情况怎么处理（站立场不明的推文放哪？）

4. **扫描范围怎么扩到合理的"全网"？**（新增 — v0.2）
   - v0.1 设计只爬 KOL 列表，v0.2 改为爬 KOL 网络的扩展节点 + 话题相关的全部 Twitter 用户
   - 但"合理边界"需要技术架构层面定义（爬多少跳？怎么避免发散到噪音区？）

5. **内容更新及时性**
   - 抓取频率、增量策略
   - "打开即新"如何保证

6. **翻译质量**
   - 翻译错了用户读了反而误导
   - 是否需要保留原文 fallback

7. **抓取方式**
   - 不用 Twitter API（用户有过决定）
   - 用什么替代

## 七、与原 KOL 项目的区别（避免重复犯错）

| 维度 | 原 KOL 项目（已凉） | 信号工作台 |
|---|---|---|
| 目的 | 自动生产 KOL 内容并发布 | 帮自己消费信息 |
| 输出方向 | 写出去（推文） | 读进来（工作台） |
| 自动化目标 | 全自动起草 + 一键发布 | 全自动整理，零交互生产 |
| AI 角色 | 内容创作者 | 翻译 / 聚类 / 筛选工具 |
| UI 复杂度 | 审核界面 + 溯源界面 + ... | 一个工作台，话题卡片为主 |
| 平台范围 | Twitter + Reddit + HN + 雪球 + 小红书 + ... | 只 Twitter（v1） |
| 自动改进系统 | L1-L6 六层自改进 | 无（v1）；v2 有"认可信号 + 历史观点验证"两条 |
| 用户角色 | 决策者（点 ✅/❌） | 阅读者 |
| 项目规模 | 极大 | 受控 |

## 八、未来路线（v2 后期，用户已认可的扩展方向）

这些机制**不在 v1 实现范围**，但是 spec 的长期方向，应作为架构选型时的**预留接口**考虑。

### 8.1 认可信号反馈

**机制**：用户在工作台里的隐式认可行为（收藏）会输入到工具的内容质量评估系统：
- 被认可的推文的来源 → 工具扫描该来源的相关内容（同作者的其他推文、被同作者引用的内容）
- 同来源被认可越多 → 内容池里该来源的权重越高

**v1 / v2 阶段衔接**（v0.3 明确）：
- **v1**：收藏数据已经在收集（推文 + 来源 + 发布时间 + 收藏时间，详见 4.5 节），但不影响排序
- **v2**：开始用收藏数据反向影响内容池权重和扫描方向

**价值**：从"用户被动消费"到"工具随用户使用而进化"。

### 8.2 历史观点验证（核心长期差异化）

**机制**：工具记录每条观点的**发布时间**。当未来事件发生后，回头看哪些观点被验证为对了 / 错了：
- 被验证对了的来源 → 权重 ↑
- 被验证错了的来源 → 权重 ↓

**这是"客观依据" — 不是 AI 给评分，是现实给评分。**

**v1 / v2 阶段衔接**（v0.3 明确）：
- **v1**：用户收藏的推文池 = 主动 curate 的"待验证观点池"。每条收藏都带发布时间。v1 阶段只收集，不验证
- **v2**：开始做事件检测和验证回放——收藏池是最干净的输入源（用户已经过滤了一遍重要性）

**输入源策略**（v0.3 修订）：
- v0.2 默认是 "AI 自动提取所有观点"，实现难度大
- v0.3 改为 **"用户收藏池优先 + AI 自动提取作为补充"**——用户主动 curate 的池子作为 v2 起步阶段的验证池，等机制稳定后再扩展到自动提取

**价值**：工具随使用时间增长而越来越准。用户的判断不再依赖"谁说话听起来对"，而依赖"谁过去说过什么、后来发生了什么"。

这是本工具相对其他信息工具（包括 Grok / ChatGPT / Twitter 原生）的**长期差异化竞争力**——其他工具没有这种"被现实验证"的反馈环。

**实现复杂度**：高。需要：
- 事件检测（如何识别"X 预测被验证"）
- 历史 → 现在的链路追踪
- （v0.3 简化）"观点提取"环节因为有用户收藏池兜底，难度降低

仍作为 v2 第一优先级。

### 8.3 其他未来扩展（右栏可挂载的面板槽位）

- "我的收藏"侧边面板（如果右栏空间允许）
- "AI 今日重点"推荐（如果未来 AI 有持久记忆能消化用户上下文）
- 跨平台扩展（Reddit / HN / 小红书等）

## 九、开放性问题（spec 之外，留给后续讨论）

1. 工作台是 Web app、桌面 app、还是本地命令行 + 浏览器渲染的混合？（**技术架构问题，用户决定**）
2. 是否支持搜索 / 历史话题查询？（v1 不做）
3. 是否需要"已读 / 未读"标记？（v1 不做，看使用后是否需要）
4. 用户 follow 列表更新后，工具是否需要"重新校准"？（v1 假设 follow 列表稳定）
5. v2 的"历史观点验证"机制 — 是否需要把"等待验证中"的观点显式标注？还是只在事后回顾时使用？
